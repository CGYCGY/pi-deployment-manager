pi-deployment-manager — architecture

A standalone agentic deployment service on pi. Any project's agent hands off a deploy over RPC; the manager owns all deployment knowledge, creds, and infra state, does the work, and returns a structured result.

sibling of pi-4b-tester · pi-e2e-tester  ·  2026-06-25  ·  realizes the pi-4b "expose hub via RPC" roadmap

1 · Topology — service, not device-driver

Client

Any project agent

Next.js / React / HTML / Astro repo. Carries zero deployment knowledge or secrets. Just sends the task.

POST /deploy
localhost HTTP · bearer token · requestId
{ project_dir, subdomain, intent }
{ status, url, health, … }
Service · gated

pi-deployment-manager

Heavily gated: no raw Bash/Edit/Read — only the 10 verbs. Wraps the 4 skill scripts, all creds, the profile registry & guards. Spawn-on-demand, runs in place on the repo.

source of truth → Coolify API Cloudflare API queried live — manager holds no persistent infra state → cold start loses nothing

Unlike the testers (hub + a persistent phone/browser spoke), a deploy runs to completion and returns — so there is no internal device-spoke.

2 · The two flows — one NL door, branch on live Coolify state

Initial first-time deploy

trigger: Coolify has no app for this project (live check; .env.deploy app-uuid is a hint)

  1. detect → pick frontend profile + backend addons
  2. scaffold deploy/Dockerfile (+ /healthz, HEALTHCHECK)
  3. convex deploy → capture prod URL (if convex)
  4. populate deploy/.env.deploy from central creds
  5. provision Coolify app + resource limits
  6. env — set vars, inject Convex URL (build-time)
  7. ⛔ subdomain-collision guard
  8. dns — create record + set Coolify domain
  9. deploy — build → GHCR → Coolify webhook
  10. ⛔ deploy-health guard → return {url}

Update redeploy after change

trigger: Coolify already has this app (live check) → API-only

  1. detect → recognized as already set up
  2. convex deploy + re-inject URL (if backend changed)
  3. redeploy — re-run deploy.sh (build → GHCR → webhook)
  4. ⛔ deploy-health guard → return

Image-based pipeline

deploy.sh builds the Docker image → pushes to GHCR (ghcr.io/<org>/<repo>) → triggers the Coolify webhook, which pulls & runs it.

4 · The gate — a heavily-gated, single-purpose agent

Skills aren't invoked by the LLM — they're the wrapped engine

A skill is a prompt that tells an LLM to run bash — exactly the capability removed. So the skill scripts become the engine the tool code calls; the model only ever sees verbs.

manager LLM ──can only call──▶
  [ detect · scaffold · provision · dns · deploy · … ]
    │ tool implementation (code)
    └─▶ coolify/tools/*.sh · cloudflare/tools/*.sh
        deploy.sh · npx convex deploy · gh

The sandbox — enforced in tool code, not trusted to the LLM

readtarget project_dir, read-only — to detect the profile
writeallowlist: deploy/* + .gitignore/.env.production + git commit. nothing else
networkCoolify · Cloudflare · GHCR · Convex · GitHub APIs
deniedgeneral Bash · Edit · Read · Glob · Write outside allowlist

This allowlist is the "only touch deploy/, never other files" guarantee.

1 purpose · 1 agent · 1 deploy at a time — the pi-e2e-tester spoke philosophy: a closed, semantic surface, not raw access.

6 · Guards — fail-closed, in code, not LLM judgment

⛔

Subdomain-collision

Caller specifies the subdomain. If it already maps to a different project on Cloudflare/Coolify → refuse. Never clobber.

🎯

Wrong-target

Every mutating Coolify/DNS call must target the app/record bound to this project (via deploy/.env.deploy UUID + domain).

❤️

Deploy-health

After ship, poll Coolify status + /healthz. Unhealthy → auto-FAIL the result. The crash-guard analog.

One server + one domain ⇒ collisions are the top silent-failure mode — hence guards live in deterministic code, not the LLM.

7 · Locked decisions at a glance

transport
Reuse pi-4b/e2e localhost-HTTP RPC (token + requestId). pi -p headless = fallback.
lifecycle
Spawn-on-demand, no daemon — deploys are infrequent; nothing to babysit.
doors
NL deploy(project_dir, subdomain, intent) first. Deterministic typed door deferred.
project access
Both modes, operate in place — no clone. Initial writes+commits; update is API-only.
subdomain
Caller-specified per deploy; collision-guarded against the single shared domain.
gating
No raw Bash/Edit/Read — only the 10 verbs. Read project / write-deploy allowlist, enforced in tool code.
execution
The 4 skill scripts are the wrapped engine the tool code calls — never invoked by the LLM.
creds
Centralized in the manager; injected into each project's gitignored deploy/.env.deploy at deploy time, never committed.
convex
Convex Cloud, deployed first; prod URL injected as build-time env into the frontend.
context reset
compact() per deploy — each deploy is an isolated task; never reset mid-flow.
stack reality
ONE Coolify VPS · ONE Cloudflare domain · every project = a subdomain on it.
